# Отчет по лабораторной работе №3 "CI/CD"

Задача лабораторной работы заключалась в написании "хорошего" и "плохого" CI/CD  
Для тестирования были использованы Мега-пет-проекты из папки [projects](../Projects) и 
"хороший" Docker [Dockerfile.good](../LabDocker/Dockerfile/Dockerfile.good)


## "Плохой" CI/CD
```commandline
name: bad CI/CD

on:
  push:
    branches:
      - '**'

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
        - name: Step 1
          uses: actions/checkout@v4

        - name: Step 2
          run: docker build -t static-site -f LabDocker/Dockerfile/Dockerfile.good .

        - name: Step 3
          run: | 
            docker run -d -p 8080:80 --name static static-site
            sleep 5
            curl --fail http://localhost:8080/landing/ChooseProject.html
```

## "Хороший" CI/CD
```commandline
name: good CI/CD

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  docker:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t static-site -f LabDocker/Dockerfile/Dockerfile.good .

      - name: Save image
        run: docker save static-site -o docker-image.tar

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: docker-image
          path: docker-image.tar

  smoke-test:
    runs-on: ubuntu-latest
    needs: docker

    steps:
      - name: Download artifact
        uses: actions/download-artifact@v4
        with:
          name: docker-image

      - name: Load image
        run: docker load < docker-image.tar

      - name: Launch container and test
        run: | 
          docker run -d -p 8080:80 --name static static-site
          timeout 15s bash -c 'until curl --fail http://localhost:8080/landing/ChooseProject.html; do sleep 1; done'
```

## Разбор полетов

### Непонятные названия шагов и jobs
Плохая практика:  
```
- name: Step 1
...
- name: Step 2
```

Хорошая практика:  
```
- name: Checkout repository
...
- name: Build Docker image
```  
Всеми любимый пример: вы откроете файл через месяц и не поймете что там происходит (ну, поймете, но это займет больше времени)
Упал ```Step3```? Удачи с разбором логов)  
Использование понятных названий сильно упрощает работу с кодом, пайплайн читается как инструкция,
также в логах GitHub Actions сразу видно, какой этап выполняется и где произошла ошибка. Это экономит время на отладку
![img.png](image/img1.png)  
![img.png](image/img2.png)  
На скринах показано, как это выглядит на гите, если в первом случае можно понять за что отвечает каждый шаг, то во втором надо заходить внутрь и пытаться разобраться

### Запуск на все подряд
Плохая практика:
```
on:
  push:
    branches:
      - '**'
```
Запускается на любой пуш в любую ветку. Исправили опечатку в `README`? Получите полную проверку работоспособности кода

Хорошая практика:  
```
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```  
Запускает пайплан только при пуше и создании PR в главную ветку.
Для простого проекта этого достаточно, но для более сложных, где несколько пайплайнов можно и нужно делать триггеры на определенные директории(например разделять тесты для фронта/бэка/сдк), чтобы
CI/CD работал только по делу и экономил время и ресурсы

### Монолитная задача
Плохая практика:
```
jobs:
  build:
```

Хорошая практика:  
```
jobs:
  docker:
  smoke-test:
```  
Почему плохо делать все в одном месте? 
1) Если пайплайн упал, то непонятно сразу на каком шаге и в чем дело, нужно слова лезть в логи и разбираться что пошло не так
2) Отсутствие гибкости, то есть, если из 5 "тестов" упал последний, нельзя перезапустить только его. Придется заново запускать всю джобу и ждать пока очередь дойдет до 5 теста, чтобы понять работает или нет

Если мы делим весь процесс на логические задачи, то получаем удобный вывод, в котором сразу понятно на каком этапе возникла проблема (если возникла)

### "Хрупкий" тест
##### Не знаю насколько это относится к CI/CD, но мне захотелось пожаловаться)
В "плохом" CI/CD было использовано `sleep 5`
для того, чтобы дать серверу время запуститься. Это не очень хорошо, так как тест становится ненадежным: если контейнер запустится за 6 секунд — тест упадет, а если за 1 секунду — мы зря прождем еще 4

В "хорошем" CI/CD мне показалось, что ждать в целом необязательно и у меня все упало:  
![img.png](image/img4.png)  
Поэтому было решено использовать умное ожидание `until curl ...`, что означает "Проверяй каждую секунду, пока не ответит", но не больше 15 секунд `timeout 15s` (для статического сайта это было бы слишком, следовательно, там все таки что-то не так)

> **PS:** Понадобилась одна секунда) пупупу

## Вывод
В ходе лабораторной работы мы на практике убедились, что рабочий CI/CD — это не всегда хороший CI/CD. 
Применение лучших практик кардинально меняет процесс разработки, экономит время и ресурсы, а также значительно упрощает отладку в случае возникновения проблем.  
Вложив немного времени в настройку правильного пайплайна один раз, мы получаем надежного помощника на все время жизни проекта, который будет беречь наши нервы и бесплатные минуты на GitHub)
